---
description: コードの変更点とドキュメントを突き合わせて、更新が必要な箇所を洗い出す
---

`develop` ブランチでの変更を `main` と比較し、`CLAUDE.md` と `docs/` 配下のドキュメントで更新が必要な箇所を洗い出してください。

## 手順

1. **変更内容を把握する**
   - `git diff main...HEAD --stat` で変更ファイル一覧を確認
   - `git diff main...HEAD` で具体的な差分を読む
   - 引数 `$ARGUMENTS` が指定されていれば、その範囲（例: 特定のコミットID、ファイルパス）に絞る

2. **影響範囲を分類する**
   変更を以下の観点で分類:
   - APIエンドポイントの追加・変更・削除 → `CLAUDE.md` の「APIエンドポイント」、`docs/02_backend_guide.md`
   - DBテーブル・カラムの変更 → `CLAUDE.md` の「データベース」、`docs/02_backend_guide.md` のモデル例
   - 認証・セキュリティ周りの変更 → `CLAUDE.md` の「認証・セキュリティ」、`docs/07_security_decisions.md`
   - AI ツール・プロンプトの変更 → `docs/04_ai_guide.md`
   - フロントエンドのページ・コンポーネント・hooks の追加 → `docs/03_frontend_guide.md`
   - パフォーマンス改善 → `docs/08_performance_optimization.md`
   - デプロイ・インフラ設定の変更 → `CLAUDE.md` の「デプロイ」
   - 完了したタスク・新しい既知の課題 → `docs/05_known_issues.md`

3. **ドキュメントを読む**
   影響しそうなドキュメントを `Read` で実際に開いて、現状の記述と差分を照合する。
   推測で「たぶんここを直す必要がある」と言わず、具体的に「N行目の○○という記述が、現在は××になっている」と指摘する。

4. **更新提案をまとめて表示する**
   以下の形式で出力:

   ```
   ## 更新が必要な箇所

   ### CLAUDE.md
   - L123: `/api/foo` のメソッドが GET → POST に変わった
   - L456: `staff_id` カラムが必須から任意になった

   ### docs/02_backend_guide.md
   - L78: clients テーブルのモデル例に新カラム xxx を追加すべき

   ## 矛盾なし
   - docs/04_ai_guide.md / docs/06_git_guide.md / docs/08_performance_optimization.md
   ```

5. **更新を実行するか確認する**
   「上記の更新を反映していい？」と聞き、ユーザーが承諾したら `Edit` で修正する。
   未経験のユーザーが学習中なので、「なぜその更新が必要か」を1行ずつ添える。

## 注意

- ドキュメントは学習ノートとしての価値もあるため、**情報を減らさない**。古い記述を新しい実装で**置き換える**ことに専念する
- コード側を正とする（ドキュメントに合わせてコードを変えない）
- 変更が大きい場合は、まずユーザーに「全部一気に直す？それともファイルごとに確認しながら直す？」と聞く

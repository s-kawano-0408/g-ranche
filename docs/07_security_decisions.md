# セキュリティ設計 — 決定事項

## 2026-03-08 決定

### リリース形態
- **クラウド公開**（3人限定：本人・母・弟）
- 他の事業所には非公開
- 3人はそれぞれ別のネットワークにいるためクラウドが必要

### 権限管理
- **管理者（admin）**: 本人のみ。利用者の登録・削除が可能。JSONマッピングを管理
- **スタッフ（staff）**: 母・弟。閲覧・記録作成・スケジュール管理が可能

## 2026-03-15 実装

### 個人情報保護方式：仮名化（SHA-256ハッシュ + JSONマッピング）

**方針：** DBに利用者の個人情報を保存しない。
受給者証番号+生年月日からSHA-256ハッシュを生成し、DBにはハッシュのみ保存。
本名とのマッピングはローカル（ブラウザのlocalStorage + JSONファイル）に保持する。

```
DB:           pseudonym_hash = "0ecb40f543205ab2" ← 個人情報なし
localStorage: {"0ecb40f543205ab2": {"family_name": "山田", ...}} ← ブラウザに保存
```

**ハッシュ生成:**
- 入力: `受給者証番号:生年月日`（例: `1311000001:1980-04-15`）
- アルゴリズム: SHA-256 → 先頭16文字に短縮
- 決定的: 同じ入力なら何度計算しても同じハッシュが出る

**仮名化対象フィールド（DBに保存しない）:**
- `family_name` / `given_name`（姓名）
- `family_name_kana` / `given_name_kana`（フリガナ）
- `birth_date`（生年月日）
- `certificate_number`（受給者証番号）

**仮名化対象外（DBにそのまま保存）:**
- `gender`, `client_type`, `staff_id`, `status`, `notes`, `end_date`

### JSONマッピングの運用
- 新規登録時 → JSONファイルが自動ダウンロード + localStorageに保存
- 管理者が母・弟にJSONファイルを共有（LINEなど）
- 母・弟は設定ページからJSONをインポート
- マッピングが未読み込みの利用者は「仮名利用者」と表示される（クリックでハッシュ表示）

### マッピング紛失時の復元
- 受給者証番号 + 生年月日 + 姓名 + フリガナを設定ページで入力
- ブラウザ内でハッシュを再計算（サーバーに個人情報を送らない）
- DB上の既存ハッシュと照合 → 一致すればマッピングを復元

### 既知のリスクと対策

| リスク | 対策 | 状態 |
|--------|------|------|
| JSONファイル紛失 | 復元機能（受給者証番号+生年月日で再生成） | 実装済み |
| localStorage消失 | JSONファイルから再インポート | 実装済み |
| 複数端末・ブラウザ | JSONファイルを管理者が共有 | 運用で対応 |
| スケジュール/記録に名前が含まれる | 運用ルール（記録に本名を書かない） | 将来の課題 |
| notesに個人情報を書く | 運用ルール | 将来の課題 |
| ハッシュ衝突 | 16文字（16進数16桁≒18京通り）で実質不可能 | 対策不要 |
